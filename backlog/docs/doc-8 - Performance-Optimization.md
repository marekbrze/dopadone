# Performance Optimization

## Overview

This document describes performance optimization strategies and implementations in the ADHD Coach project management system. It serves as a reference for optimization patterns, benchmark results, and best practices.

## Optimization Principles

1. **Measure First**: Always benchmark before optimizing to identify actual bottlenecks
2. **Server-Side Filtering**: Prefer database-level filtering over in-memory filtering
3. **Memory Efficiency**: Reduce memory allocations by loading only necessary data
4. **Query Optimization**: Use appropriate SQL features (indexes, CTEs, proper JOINs)
5. **Scalability**: Optimize for filtered result size, not total data size

## Case Studies

### Task-32: ProjectService.ListBySubareaRecursive Optimization

**Problem**: Loading ALL projects into memory and filtering in Go code was wasteful.

**Original Implementation** (In-Memory Filtering):
```go
func (s *ProjectService) ListBySubareaRecursive(ctx context.Context, subareaID string) ([]domain.Project, error) {
    // Load ALL projects (O(n) memory where n = total projects)
    allProjects, err := s.repo.ListAll(ctx)
    if err != nil {
        return nil, err
    }
    
    // Build map for O(1) parent lookups
    projectMap := make(map[string]db.Project)
    for _, p := range allProjects {
        projectMap[p.ID] = p
    }
    
    // Filter in memory
    var filtered []domain.Project
    for _, p := range allProjects {
        if belongsToSubarea(p, subareaID, projectMap) {
            filtered = append(filtered, converter.DbProjectToDomain(p))
        }
    }
    return filtered, nil
}

    // O(depth) time per project
}
```

**Issues with Original**:
- Memory: Loads O(n) records into memory (e.g., 1000 projects to show 100)
- CPU: Filters all projects in Go code
- Network: Transfers all project data from database to application
- Scalability: Performance degrades linearly with total database size

**Optimized Implementation** (Server-Side Filtering):
```go
func (s *ProjectService) ListBySubareaRecursive(ctx context.Context, subareaID string) ([]domain.Project, error) {
    // Load ONLY filtered projects (O(k) memory where k = filtered projects)
    rows, err := s.repo.ListProjectsBySubareaRecursive(ctx, sql.NullString{
        String: subareaID,
        Valid:  true,
    })
    if err != nil {
        return nil, fmt.Errorf("list projects for subarea %s: %w", subareaID, err)
    }
    
    // Convert to domain types
    projects := make([]domain.Project, len(rows))
    for i, row := range rows {
        projects[i] = converter.DbProjectRowToDomain(row)
    }
    return projects, nil
}
```

**SQL Query** (Recursive CTE):
```sql
-- name: ListProjectsBySubareaRecursive :many
WITH RECURSIVE project_hierarchy AS (
    -- Base case: projects directly in subarea
    SELECT 
        id, name, description, goal, status, priority, progress,
        deadline, color, parent_id, subarea_id, position,
        created_at, updated_at, completed_at, deleted_at
    FROM projects
    WHERE subarea_id = sqlc.narg('subarea_id') AND deleted_at IS NULL
    
    UNION ALL
    
    -- Recursive case: children of projects already in hierarchy
    SELECT 
        p.id, p.name, p.description, p.goal, p.status, p.priority, p.progress,
        p.deadline, p.color, p.parent_id, p.subarea_id, p.position,
        p.created_at, p.updated_at, p.completed_at, p.deleted_at
    FROM projects p
    INNER JOIN project_hierarchy ph ON p.parent_id = ph.id
    WHERE p.deleted_at IS NULL
)
SELECT * FROM project_hierarchy
ORDER BY position ASC, name ASC;
```

**Benefits of Optimized Version**:
- Memory: Loads only O(k) records (k = filtered results, k < n)
- CPU: Database engine performs filtering and recursion
- Network: Transfers only necessary data
- Scalability: Performance based on filtered size, not total size

## Optimization Patterns

### 1. Server-Side Filtering

**When to Use**:
- Hierarchical data with filtering requirements
- Large datasets where filtered subset << total data
- Queries with WHERE clauses that can be pushed to database

**Pattern**:
```go
// Bad: Load all, filter in memory
allData := repo.ListAll(ctx)
filtered := filterInMemory(allData, criteria)

// Good: Filter at database level
filtered := repo.ListWithFilter(ctx, criteria)
```

**Benefits**:
- Reduces memory footprint
- Reduces network transfer
- Leverages database query optimization

### 2. Recursive CTEs for Hierarchies

**When to Use**:
- Tree-structured data (projects, categories, etc.)
- Need to traverse parent-child relationships
- Filtering required at multiple hierarchy levels

**Pattern**:
```sql
WITH RECURSIVE hierarchy AS (
    -- Base case
    SELECT * FROM table WHERE base_condition
    
    UNION ALL
    
    -- Recursive case
    SELECT t.* FROM table t
    INNER JOIN hierarchy h ON t.parent_id = h.id
)
SELECT * FROM hierarchy;
```

**Benefits**:
- Single query instead of multiple round trips
- Database engine handles recursion efficiently
- Automatic cycle detection (SQLite)

### 3. Proper Indexing

**Current Indexes**:
```sql
-- Projects table
CREATE INDEX idx_projects_deadline ON projects(deadline);
CREATE INDEX idx_projects_status_priority ON projects(status, priority);
CREATE INDEX idx_projects_parent_id ON projects(parent_id);
CREATE INDEX idx_projects_subarea_id ON projects(subarea_id);

-- Subareas table
CREATE INDEX idx_subareas_area_id ON subareas(area_id);
```

**Guidelines**:
- Index columns used in WHERE clauses
- Index foreign keys (parent_id, subarea_id, etc.)
- Composite indexes for multi-column filters (status + priority)
- Measure impact before/after adding indexes

## Benchmarking Strategy

### Running Benchmarks

```bash
# Run specific benchmark
go test -bench=BenchmarkListBySubareaRecursive -benchmem

# Run all benchmarks
go test -bench=. -benchmem

# With CPU profiling
go test -bench=. -cpuprofile=cpu.prof
go tool pprof cpu.prof

# With memory profiling
go test -bench=. -memprofile=mem.prof
go tool pprof mem.prof
```

### Interpreting Results

```
BenchmarkListBySubareaRecursive/size=100-8   	  845678	      1342 ns/op	    8608 B/op	      22 allocs/op
```

- `845678`: Number of iterations
- `1342 ns/op`: Nanoseconds per operation (lower is better)
- `8608 B/op`: Bytes allocated per operation (lower is better)
- `22 allocs/op`: Number of allocations per operation (lower is better)

### Benchmark Best Practices

1. **Test Multiple Sizes**: Benchmark with small, medium, and large datasets
2. **Use Realistic Ratios**: Test with realistic filter ratios (e.g., 10%, 20%)
3. **Reset Timer**: Use `b.ResetTimer()` after setup, before measurement
4. **Avoid Compiler Optimizations**: Use `b.N` in computations or assign to globals
5. **Measure Allocations**: Always use `-benchmem` flag

## Performance Monitoring

### Key Metrics to Track

1. **Response Time**: Time to complete operations
2. **Memory Usage**: Heap allocation size
3. **Allocation Count**: Number of memory allocations
4. **Database Query Time**: Time spent in database queries
5. **Data Transfer**: Amount of data transferred from database

### Profiling Tools

```bash
# CPU profiling
go tool pprof http://localhost:6060/debug/pprof/profile

# Heap profiling
go tool pprof http://localhost:6060/debug/pprof/heap

# Goroutine profiling
go tool pprof http://localhost:6060/debug/pprof/goroutine

# Trace

```

## Future Optimization Opportunities

### 1. Connection Pooling

**Current**: Each command opens/closes database connection
**Opportunity**: Use connection pool for better concurrency

### 2. Prepared Statements

**Current**: sqlc generates queries but doesn't cache prepared statements
**Opportunity**: Cache prepared statements for repeated queries

### 3. Query Result Caching

**Use Case**: Frequently accessed, rarely changed data
**Implementation**: In-memory cache with TTL

### 4. Batch Operations

**Use Case**: Bulk creates/updates
**Implementation**: Use transactions with batch INSERT/UPDATE

### 5. Pagination

**Use Case**: Large result sets
**Implementation**: Add LIMIT/OFFSET or cursor-based pagination

### 6. Lazy Loading

**Use Case**: Related entities not always needed
**Implementation**: Load related data on demand

- [ ] Plan for measuring improvement

- [ ] Run benchmarks to verify improvement

- [ ] Update documentation
