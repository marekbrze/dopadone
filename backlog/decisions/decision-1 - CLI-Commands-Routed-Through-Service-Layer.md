# CLI Commands Routed Through Service Layer

## Context

Initially, CLI commands in `cmd/projectdb/projects.go` and `cmd/projectdb/tasks.go` directly used the `db.Querier` interface generated by sqlc. This created several issues:

1. **Business Logic Duplication**: Validation and business rules were scattered across CLI command handlers
2. **Direct Database Coupling**: CLI layer was tightly coupled to database implementation details
3. **Type Safety Issues**: Working with sqlc's `sql.NullString` types was error-prone
4. **Limited Reusability**: Logic couldn't be reused by TUI or future API layer
5. **Testing Complexity**: Testing CLI commands required full database setup

## Decision

We refactored all CLI commands to route through a dedicated service layer:

1. **Created Service Layer**: Implemented `ProjectService`, `TaskService`, `SubareaService`, and `AreaService` in `internal/service/`
2. **Introduced Domain Types**: Created proper domain types in `internal/domain/` (e.g., `TaskStatus`, `Priority`, `ProjectStatus`)
3. **Added Service Container**: Created `ServiceContainer` in `main.go` to manage service instances
4. **Implemented Converters**: Added DB → Domain converters in `internal/converter/`
5. **Refactored CLI Commands**: Updated all CRUD operations to use services instead of direct DB access

## Implementation Details

### Before

```go
func runProjectsCreate(cmd *cobra.Command, args []string) {
    db, err := GetDB()
    if err != nil {
        cli.ExitWithError(err)
    }
    defer db.Close()
    
    queries := db.New(db)
    project, err := queries.CreateProject(ctx, db.CreateProjectParams{
        Name:      projectName,
        Status:    "active",
        Priority:  "medium",
        // sql.NullString types for optional fields
        Color:     sql.NullString{String: projectColor, Valid: true},
    })
}
```

### After

```go
func runProjectsCreate(cmd *cobra.Command, args []string) {
    services, err := GetServices()
    if err != nil {
        cli.ExitWithError(err)
    }
    defer services.Close()
    
    params := service.CreateProjectParams{
        Name:     projectName,
        Status:   domain.ProjectStatusActive,
        Priority: domain.PriorityMedium,
        Color:    domain.Color(projectColor),
    }
    
    project, err := services.Projects.Create(ctx, params)
}
```

### Service Container Pattern

```go
type ServiceContainer struct {
    Projects *service.ProjectService
    Tasks    *service.TaskService
    Subareas *service.SubareaService
    Areas    *service.AreaService
    db       *sql.DB
}

func GetServices() (*ServiceContainer, error) {
    dbConn, err := GetDB()
    if err != nil {
        return nil, err
    }
    queries := db.New(dbConn)
    
    return &ServiceContainer{
        Projects: service.NewProjectService(queries),
        Tasks:    service.NewTaskService(queries),
        Subareas: service.NewSubareaService(queries),
        Areas:    service.NewAreaService(queries),
        db:       dbConn,
    }, nil
}
```

## Consequences

### Positive

1. **Clean Architecture**: Clear separation between presentation and business logic
2. **Type Safety**: Domain types prevent invalid states at compile time
3. **Testability**: Services can be mocked for unit testing without database
4. **Reusability**: Service layer can be used by CLI, TUI, and future REST API
5. **Validation**: Centralized business validation in one place
6. **Error Handling**: Domain-specific errors with proper context
7. **Maintainability**: Changes to business logic don't affect CLI layer

### Negative

1. **Additional Abstraction**: One more layer to understand for new developers
2. **More Code**: Service layer adds code that wasn't there before
3. **Type Conversion**: Need to convert between DB types and domain types

### Neutral

1. **Performance**: Minimal overhead from additional abstraction layer
2. **File Count**: More files but better organized

## Alternatives Considered

1. **Keep Direct DB Access**: Would maintain simplicity but perpetuate all existing issues
2. **Repository Pattern**: More abstraction than needed for this use case
3. **CQRS**: Over-engineering for current requirements
4. **ORM (GORM, etc.)**: Would replace sqlc but adds runtime overhead and less type safety

## Migration Path

The refactoring was done incrementally:

1. **Phase 1**: Create service layer infrastructure (ServiceContainer, domain types, converters)
2. **Phase 2**: Refactor `projects.go` commands
3. **Phase 3**: Refactor `subareas.go` commands
4. **Phase 4**: Verify `areas.go` already using services
5. **Phase 5**: Refactor `tasks.go` commands
6. **Phase 6**: Remove helper functions and verify all tests pass
